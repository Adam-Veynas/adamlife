# AdamLife Design System

## Purpose

This document defines the visual design principles of AdamLife.

The goal is to create a clean, modern and consistent interface that feels fast, intuitive and comfortable for everyday use.

Every screen should follow the same design language.

---

# Design Philosophy

AdamLife is designed around simplicity.

The interface should never distract the user.

Visual hierarchy must always be clear.

Users should immediately understand:

- where they are
- what they can do
- what is important

---

# Design Principles

The interface must always be:

- Clean
- Minimal
- Fast
- Consistent
- Functional

Every visual element must have a purpose.

Decorative elements should be kept to a minimum.

---

# Color Palette

Use a limited color palette.

Primary color:
- Blue

Secondary color:
- Neutral Gray

Background:
- Light Gray or White

Success:
- Green

Warning:
- Orange

Error:
- Red

Avoid excessive color usage.

Colors should communicate meaning rather than decoration.

---

# Typography

Text should always remain readable.

Use clear hierarchy:

- Large Title
- Page Title
- Section Title
- Body Text
- Caption

Avoid unnecessary font variations.

Consistency is more important than creativity.

---

# Spacing

Use consistent spacing throughout the application.

Recommended spacing values:

- 4
- 8
- 12
- 16
- 24
- 32

Avoid random margins and padding.

---

# Border Radius

Rounded corners should remain consistent.

Standard radius:

- Small
- Medium
- Large

Do not mix multiple corner styles on the same screen.

---

# Icons

Use a single icon family across the application.

Icons should always have a clear meaning.

Avoid decorative icons.

---

# Cards

Cards are the primary content container.

Each card should contain one logical piece of information.

Cards should:

- have comfortable padding
- use rounded corners
- avoid unnecessary borders

---

# Buttons

Buttons must clearly communicate actions.

Primary button:

Used for the main action.

Secondary button:

Used for less important actions.

Danger button:

Used only for destructive operations.

---

# Navigation

Navigation should remain simple.

Bottom navigation contains the main modules.

Avoid deep navigation whenever possible.

Users should always know how to return.

---

# Forms

Forms should remain short and simple.

Every field should include:

- label
- validation
- helpful error message

Avoid unnecessary required fields.

---

# Lists

Lists should remain easy to scan.

Each list item should have clear spacing.

Important information should stand out naturally.

---

# Animations

Animations should feel smooth.

Use animations only when they improve user experience.

Avoid excessive motion.

Speed should feel natural.

---

# Loading States

Never leave the user wondering.

Show loading indicators whenever data is being processed.

Avoid blocking the entire interface when unnecessary.

---

# Empty States

Every empty screen should explain:

- why it is empty
- what the user can do next

Avoid blank screens.

---

# Error States

Errors should be understandable.

Avoid technical messages.

Always explain the problem in simple language.

Whenever possible, provide a recovery action.

---

# Accessibility

Maintain sufficient contrast.

Buttons should be easy to tap.

Text should remain readable.

Avoid relying only on color to communicate information.

---

# Dashboard Philosophy

The Dashboard is the application's home.

It should answer one question:

"What do I need to know today?"

Only the most important information belongs here.

---

# Finance Philosophy

Finance screens should focus on clarity.

Income, expenses and balances must be understandable at a glance.

Avoid unnecessary charts.

Information should be actionable.

---

# Planner Philosophy

Planner screens should encourage completing tasks.

The interface should reduce mental load.

Tasks should always be easy to create, edit and complete.

---

# Vehicle Philosophy

Vehicle information should focus on maintenance.

Important dates and mileage should always be visible.

The user should never forget upcoming service.

---

# Consistency Rule

If two screens solve similar problems, they should look and behave similarly.

Consistency always wins over uniqueness.

---

# Design Principles Summary

Every design decision should satisfy at least one of these goals:

- Improve clarity
- Reduce complexity
- Increase speed
- Improve usability
- Maintain consistency

If it satisfies none of them, it should not be added.
